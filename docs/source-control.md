# Source Control Integration

How to set up CodeFlow so agents can clone repos, edit files, commit, push,
and open PRs or MRs.

## 1. Pick a host (one per instance)

A CodeFlow instance is configured for **exactly one** git host. Switch modes
by re-entering settings; tokens for the previous mode are discarded.

- **GitHub mode** — talks to `github.com`. You supply one personal access
  token.
- **GitLab mode** — talks to a self-hosted GitLab (e.g. `https://gitlab.example.com`).
  You supply the base URL and one personal access token.

Attempting to `workspace.open` a URL whose host doesn't match the configured
mode is rejected by `RepoUrlHostGuard` before anything touches the filesystem
or network.

## 2. Create a token with the right scopes

- **GitHub**: a classic PAT or fine-grained token with at minimum `repo`
  (contents + pull requests). If the scratch repos the agent will use live
  under an org, grant access to that org.
- **GitLab**: a personal access token with at minimum `api` and
  `write_repository`.

Copy the token value when you generate it — neither host will show it again
and CodeFlow will not surface it back to you after you save it.

## 3. Configure the host in CodeFlow

Navigate to **Settings → Git Host** (`/settings/git-host`). This route is
guarded by the `git_host:write` permission, which the default `admin` role
carries.

1. Pick **GitHub** or **GitLab**.
2. If GitLab, enter the base URL (no trailing slash; `/api/v4` is appended
   by CodeFlow when it talks to GitLab).
3. Paste the token into the password field.
4. Click **Save**.
5. Click **Verify connection** — this calls `GET /user` on GitHub or
   `GET /api/v4/user` on GitLab with the stored token and records the
   timestamp on success.

The token is encrypted at rest via the Phase A `AesGcmSecretProtector`. API
`GET` responses return only a `hasToken` boolean; the token never leaves the
server after you save it.

Under the hood:

- `PUT /api/admin/git-host` — upserts mode + baseUrl + token.
- `GET /api/admin/git-host` — returns `{ mode, baseUrl, hasToken, lastVerifiedAt }`.
- `POST /api/admin/git-host/verify` — live connectivity check, records
  `lastVerifiedAt` on success.

## 4. Grant tools to roles

Tool access is role-based (Phase B). Navigate to **Settings → Roles** and
pick or create a role, then tick the relevant tools in the picker.

| Tool | Category | Mutating | Notes |
| --- | --- | --- | --- |
| `workspace.open` | Host | no | Opens (or reuses) a workspace for a repo URL inside the current correlation. |
| `workspace.list_files` | Host | no | Lists files under the workspace (skips `.git`). |
| `workspace.read_file` | Host | no | Reads a file capped at `WorkspaceOptions.ReadMaxBytes` (default 512 KiB). |
| `workspace.write_file` | Host | yes | Writes/overwrites UTF-8 content, path-confined to the workspace. |
| `workspace.delete_file` | Host | yes | Deletes a single file, path-confined. |
| `workspace.exec` | Host | yes | **HIGH RISK** — runs an executable in the workspace. See `SECURITY.md`. |
| `vcs.create_branch` | Host | yes | Creates + checks out a work branch (default `codeflow/<correlation[:8]>/<slug>`). |
| `vcs.commit` | Host | yes | `git add -A` + commit. Returns `committed=false` if there's nothing to commit. |
| `vcs.push` | Host | yes | Pushes the current branch to origin. Rejects the default branch. No force-push. |
| `vcs.open_pr` | Host | yes | Opens a PR (GitHub) or MR (GitLab). |

Suggested role bundles:

- **Workspace reader** — `workspace.open`, `workspace.list_files`, `workspace.read_file`.
- **Workspace editor** — reader plus `workspace.write_file`, `workspace.delete_file`.
- **Full VCS** — editor plus all `vcs.*` tools.
- **Builder** — Full VCS plus `workspace.exec`. **Only grant on instances that serve trusted agent authors.**

Grants are per-role; agents pick up the changes on their next invocation
without an agent-version bump.

## 5. Usage sketch for an agent workflow

A prompt template for an agent granted the Full-VCS bundle might drive:

```
1. workspace.open({ "repoUrl": "https://github.com/acme/widget.git" })
   → { repoSlug: "acme-widget", defaultBranch: "main", currentBranch: "codeflow/wt/…" }

2. vcs.create_branch({ "repoSlug": "acme-widget", "name": "feature/add-hello" })

3. workspace.write_file({
     "repoSlug": "acme-widget",
     "path": "docs/hello.md",
     "content": "# hello\n"
   })

4. vcs.commit({ "repoSlug": "acme-widget", "message": "add hello doc" })
   → { committed: true, sha: "a1b2c3…" }

5. vcs.push({ "repoSlug": "acme-widget" })

6. vcs.open_pr({
     "repoSlug": "acme-widget",
     "title": "Add hello doc",
     "body": "Generated by the docs-agent."
   })
   → { url: "https://github.com/acme/widget/pull/42", number: 42, head: "feature/add-hello", base: "main" }
```

## 6. Lifecycle & cleanup

- Every workflow-terminating transition (Completed / Failed / Escalated)
  publishes `WorkflowTerminated`, and `WorkflowTerminatedConsumer` calls
  `WorkspaceService.ReleaseAsync` to delete the correlation's worktrees.
- Worktrees orphaned by a process crash are swept hourly (and once at
  startup) by `WorktreeSweepHostedService`. Anything in
  `{workspace-root}/work/<correlation>/` whose mtime exceeds
  `WorkspaceOptions.WorktreeTtl` (default 24h) is removed; mirrors are
  then pruned.
- The bare-mirror cache under `{workspace-root}/cache/` is kept
  indefinitely — hitting a repo twice skips the clone.

## 7. Observability

- Traces: every VCS provider call and every tool invocation emits an
  OpenTelemetry span; tokens are never added as span attributes.
- Disk: `GET /api/ops/workspace` returns `{ rootBytes, cacheBytes,
  worktreeCount, warnThresholdBytes, aboveWarn }`. Thresholds configured
  via `WorkspaceOptions.DiskUsageWarnBytes`.
- Metrics: `/api/ops/metrics` exposes `codeflow_workspace_bytes` and
  `codeflow_workspace_worktree_count` Prometheus gauges alongside the
  existing DLQ gauges.

## 8. Known risks

Read `SECURITY.md` in the repository root for the full threat model.
Highlights:

- Do not grant `workspace.exec` on instances serving untrusted authors.
  It runs agent-directed commands on the CodeFlow host with no sandbox in v1.
- Push auth injects the token via a per-process `-c http.extraheader`
  argument. It is not written to disk, but is visible in `ps` output for
  the duration of the git invocation.
- No multi-account support: one GitHub token **or** one GitLab token
  per instance.

## 9. Configuration reference

`CodeFlow:Workspace` options (from `WorkspaceOptions`):

| Setting | Default | Purpose |
| --- | --- | --- |
| `Root` | `{cwd}/workspace` | Root for `cache/` and `work/`. Must be writable. |
| `WorktreeTtl` | `24h` | Orphan worktree cutoff for the periodic sweep. |
| `ReadMaxBytes` | `524288` (512 KiB) | `workspace.read_file` cap. |
| `ExecTimeoutSeconds` | `600` | Per-call timeout for `workspace.exec`. |
| `ExecOutputMaxBytes` | `1048576` (1 MiB) | Output cap per stream for `workspace.exec`. |
| `GitCommandTimeout` | `10m` | Per-invocation timeout for the git CLI. |
| `ExecEnvAllowlist` | `PATH, HOME, LANG, LC_ALL, USER, TMPDIR, SHELL` | Env vars that cross into `workspace.exec` children. |
| `DiskUsageWarnBytes` | `10 GiB` | Threshold for the `aboveWarn` flag on `/api/ops/workspace`. |
| `DiskUsageCacheDuration` | `60s` | How long `IWorkspaceUsageReporter` caches a directory walk. |
