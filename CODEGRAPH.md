## Summary

CodeFlow is a full-stack AI agent workflow orchestration platform composed of an Angular frontend (codeflow-ui) and a layered .NET backend. The system allows users to design, execute, monitor, and replay multi-agent AI workflows through a visual canvas editor. The backend is structured into well-separated layers: CodeFlow.Api (ASP.NET Core REST API and SSE streaming entry point), CodeFlow.Orchestration (MassTransit-based saga state machine driving multi-step, multi-agent workflow execution including subflows, swarm parallelism, human-in-the-loop tasks, review loops, and notification dispatch), CodeFlow.Runtime (core agent invocation engine integrating LLM providers — OpenAI, Anthropic, LM Studio — alongside tool providers for Docker, filesystem, web fetch/search, and MCP protocol servers, plus Git-backed workspace management), CodeFlow.Persistence (EF Core data access layer over MariaDB covering all domain entities), CodeFlow.Host (DI wiring, background services, and concrete infrastructure implementations bridging Runtime abstractions to real dependencies), CodeFlow.Contracts (shared interfaces and DTOs for notification and messaging infrastructure enabling dependency inversion), and CodeFlow.Worker (background worker hosting long-running services). Architectural correctness is enforced at the source level by CodeFlow.Architecture.Tests, while CodeFlow.Api.Tests, CodeFlow.Orchestration.Tests, CodeFlow.Runtime.Tests, CodeFlow.Persistence.Tests, and CodeFlow.Contracts.Tests provide integration and unit test coverage, many using Testcontainers to spin up real MariaDB and RabbitMQ instances. Externally, CodeFlow makes HTTP calls to a CodeGraph service (likely for code analysis or graph queries) and exchanges messages bidirectionally with a 'codeflow-team' system via a publish/subscribe mechanism (RabbitMQ), indicating it operates as part of a broader distributed platform where team-level events and workflow events flow in both directions. Key infrastructure dependencies include MariaDB, RabbitMQ (via MassTransit), Docker, and optionally GitHub/GitLab for VCS integration. Scripting within workflows is provided by Jint (JavaScript) and Scriban (templating).

## Projects

### CodeFlow

CodeFlow is a full-stack workflow orchestration platform consisting of an Angular frontend (codeflow-ui) and a .NET backend. The UI allows users to design, execute, monitor, and replay multi-agent AI workflows through a visual canvas editor, with support for agent configuration, LLM provider management, MCP server integration, notification routing, and trace/audit tooling. The backend handles orchestration, persistence, runtime execution of agents, authorization, and messaging infrastructure.

### CodeFlow.Api

CodeFlow.Api is the ASP.NET Core web API layer for the CodeFlow platform, which orchestrates AI agent workflows. It exposes REST endpoints for managing agents, workflows, traces, roles, MCP servers, notifications, and an AI assistant, while also handling authentication/authorization, workflow validation, package import/export, and real-time trace event streaming via SSE. It acts as the primary entry point that wires together persistence, runtime orchestration, and LLM provider integrations (OpenAI, Anthropic).

### CodeFlow.Api.Tests

CodeFlow.Api.Tests is the integration and unit test project for the CodeFlow API. It covers HTTP endpoint behaviour, workflow template materialization, workflow validation rules, assistant tool invocations, workflow package import/export, dead-letter queue handling, MCP server connectivity, and authentication — using Testcontainers to spin up real MariaDB and RabbitMQ instances for end-to-end scenarios alongside lightweight in-memory fakes and stubs for isolated unit tests.

### CodeFlow.Architecture.Tests

CodeFlow.Architecture.Tests is an architectural enforcement test project that uses source-code scanning to verify layering and boundary rules across the CodeFlow repository. It scans source files for forbidden patterns (such as provider SDK imports, network/filesystem mutations, or merge operations) in specific project layers, ensuring that architectural constraints like dependency direction and authority boundaries are upheld at the code level rather than just at runtime.

### CodeFlow.Contracts

CodeFlow.Contracts defines the shared abstractions and data transfer types for the CodeFlow system's notification infrastructure. It provides interfaces and result types that decouple notification dispatching, event modeling, and provider delivery from their concrete implementations, which reside in other projects (e.g., CodeFlow.Orchestration). This project acts as a contracts/interfaces layer, enabling dependency inversion across the repository.

### CodeFlow.Contracts.Tests

CodeFlow.Contracts.Tests is a test project that verifies the serialization and deserialization behavior of contract types defined in the CodeFlow.Contracts library. It validates JSON round-trip fidelity, default value handling, and structural correctness for messaging contracts related to subflow orchestration, agent invocations, and human-in-the-loop (HITL) notification workflows.

### CodeFlow.Host

CodeFlow.Host is the infrastructure and hosting layer of the CodeFlow system. It provides dependency injection registrations, background services, and concrete implementations of runtime interfaces for concerns such as LLM provider configuration, dead-letter queue management, workspace/VCS setup, observability, container resource sweeping, and data cleanup. It bridges the abstract CodeFlow.Runtime contracts with real external dependencies (RabbitMQ, databases, container tooling, and cloud/local LLM providers).

### CodeFlow.Orchestration

CodeFlow.Orchestration is the workflow execution engine of the CodeFlow system. It implements a MassTransit-based saga state machine that drives multi-step, multi-agent workflows: consuming agent invocation requests, routing decisions through workflow edges, managing review loops and subflows, coordinating swarm-style parallel agent invocations, handling human-in-the-loop (HITL) tasks, and dispatching notifications via email, Slack, and SMS. It also provides a dry-run simulation mode, replay/diff tooling, and scripting infrastructure (Jint-based logic nodes, Scriban templates) for workflow dataflow analysis and output transformation.

### CodeFlow.Orchestration.Tests

CodeFlow.Orchestration.Tests is the integration and unit test project for the CodeFlow orchestration layer. It exercises workflow saga state machines, agent invocation pipelines, replay mechanics, notification dispatch, scripting/transform nodes, swarm coordination patterns, and token usage tracking. The tests range from in-process unit tests with fakes and stubs to full end-to-end integration tests that spin up real MariaDB and RabbitMQ instances via Testcontainers.

### CodeFlow.Persistence

CodeFlow.Persistence is the data access layer for the CodeFlow system, providing Entity Framework Core-based repositories, database migrations, and related abstractions for all persistent domain entities. It manages storage and retrieval of workflows, agent configurations, roles, skills, MCP servers, LLM provider settings, assistant conversations, notification infrastructure, artifact blobs, token usage records, and authority/refusal audit data. It exposes repository interfaces consumed by orchestration, API, and runtime components throughout the repository.

### CodeFlow.Persistence.Tests

CodeFlow.Persistence.Tests is an integration test project for the CodeFlow.Persistence layer, validating that repositories, entity mappings, encryption, and domain services behave correctly against a real MariaDB database. Every test class spins up an ephemeral MariaDB 11.4 container via Testcontainers, runs EF Core migrations, exercises the production repository and service code, and then tears the container down. The project covers agents, roles, skills, MCP servers, workflows, sagas, notifications, conversation history, authority/refusal events, secret protection, artifact storage, and token usage tracking.

### CodeFlow.Runtime

CodeFlow.Runtime is the core runtime library for an AI agent orchestration system. It provides the fundamental abstractions and implementations for invoking LLM model clients (OpenAI, Anthropic, LM Studio), managing tool providers (workspace file operations, Docker container execution, web fetching/search, MCP protocol servers), assembling prompt context via Scriban templates, enforcing access policies and admission controls, and managing Git-backed workspaces with VCS provider integrations (GitHub, GitLab). It acts as the central engine that other projects (API, Orchestration, Persistence, Host) depend on for agent invocation, tool execution, and workspace management.
